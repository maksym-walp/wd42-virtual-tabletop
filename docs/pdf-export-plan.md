# Експорт листа персонажа в PDF — план реалізації

> Цей файл — самодостатній опис задачі, придатний як промпт для майбутньої реалізації (Claude Code або будь-кого іншого). Містить контекст, аналіз варіантів, рекомендацію та конкретний план файлів. Нічого з цього ще не реалізовано.

## 1. Задача

Додати можливість завантажити лист персонажа як PDF-файл, придатний для друку. PDF має виглядати **не як сторінка сайту** (не таби, не мобільно-компактні картки), а як щільний однопосторінковий (чи кількасторінковий) друкований аркуш — той самий стиль, що вже присутній у референсних зображеннях `ttrpg-system/Боєць.png`, `ttrpg-system/Чаклун.png`, `ttrpg-system/Пройдисвіт.png` (надані користувачем): усі секції видно одразу — характеристики з кубиками-модифікаторами, здоров'я (сітка кубиків), чари (кола енергії + список заклинань), бойове спорядження (таблиця), монети (5 монетних дворів × 2 номінали), нотатки.

Бажано (nice-to-have, уточнити обсяг під час реалізації): можливість також експортувати списки заклинань і "маневрів" окремо або в межах того самого документа.

PDF **не обов'язково має лишатись редагованим/заповнюваним формою** — досить, щоб він гарно друкувався. Це важливо: знімає вимогу робити AcroForm-поля.

## 2. Розглянуті варіанти та чому обрано Puppeteer

### Варіант A — HTML→PDF через headless-браузер (Puppeteer/Playwright) ✅ РЕКОМЕНДОВАНО
Рендеримо окрему, спеціально зверстану HTML/CSS-сторінку (React-компонент) і перетворюємо її на PDF через `page.pdf()` у headless Chromium.

**За:**
- `services/frontend/src/pages/CharacterSheet.jsx` вже писався з оглядкою на друкований лист (в коді є коментарі на кшталт `{/* Banner (matches the PDF top row) */}`, `{/* Dice pool — matches PDF cells */}`) — кольори/шрифти/тема (`ink-on-parchment`, Cinzel/Lora) вже готові й підтримуються.
- Гра активно розвивається (система грошей щойно додана, дерево розвитку "в розробці до v0.6" за словами самої книги правил) — HTML-шаблон живе в тому ж React/JS коді й лишається синхронним з рештою застосунку. Будь-яка нова механіка одразу доступна для PDF без окремої ручної роботи.
- Повний контроль над CSS-пагінацією: `@page { size: A4; margin: ... }`, `break-inside: avoid` на секціях, multi-page — саме те, що треба для щільного багатосекційного макету.
- Немає нової інфраструктурної залежності поза Node.js-екосистемою, яку вже використовує весь проєкт.

**Проти:**
- Puppeteer тягне за собою Chromium (~300МБ) — треба акуратно налаштувати в Docker-образі (Alpine-базовий образ погано дружить із бандлованим Chromium з коробки, потрібен `puppeteer-core` + системний Chromium через `apk add chromium`, або перехід на Debian slim-базу для цього сервісу).

### Варіант B — Заздалегідь підготовлений PDF-шаблон (Adobe Acrobat/LibreOffice) + заповнення полів (`pdf-lib` у Node, або `fillpdf`/`PyPDFForm` у Python)
Дизайнер вручну верстає PDF з іменованими AcroForm-полями в спеціалізованому ПЗ, код лише підставляє значення в готові поля.

**За:** легка залежність (`pdf-lib` — чистий JS, без Chromium), пікселево точний друкований вигляд незалежно від рендерера браузера.

**Проти:** два паралельні джерела правди — React-UI і окремий PDF-шаблон. Кожна зміна механіки (нове поле, нова секція) вимагає ручного редагування PDF-шаблону в іншому інструменті й ручного перепідключення імен полів у коді. При активно змінюваній грі (як зараз) це стає постійним тягарем. **Відхилено з цієї причини.**

### Варіант C — Експорт у JSON + сторонній сервіс/окремий сайт "JSON → PDF"
Віддати JSON через уже наявний `GET /api/characters/:id`, конвертацію в PDF доручити зовнішньому інструменту чи окремому сайту.

**За:** знімає роботу з цього проєкту.

**Проти:**
- Дані персонажів (потенційно приватні) ідуть на сторонній сервіс — небажано для self-hosted проєкту, де вже є власний бекенд і авторизація.
- Все одно треба десь верстати шаблон (у форматі того сервісу) — та сама проблема синхронізації, що й у варіанті B, тільки в чужому інструменті.
- Зайвий крок для користувача (експортувати JSON → перенести на інший сайт), замість однієї кнопки "Завантажити PDF" просто на сторінці персонажа.
- Генералізований зовнішній інструмент однаково не знатиме специфічної механіки цієї гри (монети, архетипи, дерево розвитку) — переваги "готового рішення" тут немає.

**Відхилено.**

### Побіжно розглянутий варіант — `@react-pdf/renderer`
Пише PDF-макет як React-компоненти (`<Page>`, `<View>`, `<Text>`) без headless-браузера взагалі — значно легша залежність (кілька МБ, без Chromium), швидший рендер, працює і на клієнті, і на сервері. Мінус — стилізація описується окремим API (схожим на React Native `StyleSheet`), Tailwind-класи напряму не переносяться, отже верстку секцій довелось би написати заново в цій системі. Користувач вже висловив перевагу до Puppeteer, тож це лишається лише як нотатка "якщо розмір Docker-образу через Chromium стане реальною проблемою — можна пізніше мігрувати на `@react-pdf/renderer` замість Puppeteer, зберігши ту саму ідею окремого друкованого макету".

## 3. Архітектура (рекомендована)

### 3.1. Окремий "друкований" макет на фронтенді
Новий файл `services/frontend/src/pages/CharacterSheetPrint.jsx`, новий маршрут `/characters/:id/print` (додати в `services/frontend/src/App.jsx`).

Це **не** повторне використання табів з `CharacterSheet.jsx` — це новий, щільний, все-на-одному-екрані макет, натхненний `ttrpg-system/*.png`:
- Шапка: ім'я, архетип/раса, коло під портрет (заглушка, як "Портрет або символ" на референсі).
- Банер-статистика: натхнення / захист / героїчні дії / ініціатива (переюзати `BannerBox`-подібну верстку з `CharacterSheet.jsx`).
- 5 груп характеристик (`CHARACTERISTICS` з `constants/characterSheet.js`) — навички з кубиками-модифікаторами, рівень характеристики.
- Здоров'я: сітка кубиків (`health_dice_values`), шкала смерті (`death_scale`), стани (`conditions`).
- Бойове спорядження: таблиця (переюзати структуру з `EquipmentTab`, `CharacterSheet.jsx`).
- Чари: кола магічної енергії (`current_magic`/`maxMagic`) + список відомих заклинань (`known_spells` + дані з `spellbook`-сервісу — уже підвантажуються в `CharacterSheet.jsx` через `spellbookApi`).
- Монети: 5 монетних дворів × 2 номінали (`CURRENCIES` з `constants/characterSheet.js`, поле `money` на персонажі).
- Нотатки (`notes`/`backstory`).
- "Маневри" — **уточнити під час реалізації**: окремої структурованої сутності "маневр" у поточній схемі БД, здається, немає (є вільне поле `notes` на предметах спорядження й вільний текст нотаток персонажа). Якщо так лишиться — вивести як довільний текстовий блок, а не динамічний список. Якщо до моменту реалізації з'явиться окрема механіка маневрів — підключити її так само, як список заклинань.

CSS: `@media print` + `@page { size: A4; margin: 12mm; }`, `break-inside: avoid` на кожній секції-картці, щоб секція не розрізалась між сторінками невдало.

Ця сторінка має вміти рендеритись **без табів/навбару/боттомнаву** — окремий "голий" layout, не обгорнутий у звичайний `Navbar`/`BottomNav` shell (перевірити, як зараз влаштований `App.jsx` — на момент запису цього файлу там `<div className="flex h-dvh flex-col"><Navbar/><div class="flex-1 overflow-y-auto"><Routes/></div><BottomNav/></div>`; маршрут `/characters/:id/print`, ймовірно, варто винести ПОЗА цю обгортку, як окремий top-level `<Route>` без Navbar/BottomNav).

### 3.2. Генерація PDF через Puppeteer на бекенді
Вбудувати в наявний сервіс `services/character-sheet` (не заводити окремий мікросервіс — менше рухомих частин; якщо розмір Docker-образу через Chromium стане проблемою, винести пізніше в окремий сервіс `pdf-export`).

- Новий ендпоінт: `GET /api/characters/:id/pdf`.
- Права доступу — ті самі, що в `getSheet`/`getPublicSheet` (власник, `game_master`, або `is_public`).
- Бекенд:
  1. Перевіряє права доступу як у `getSheet`.
  2. Генерує короткоживучий "print-токен" (підписаний JWT, TTL кілька хвилин, у payload — `characterId` і, можливо, `userId` для повторної перевірки прав).
  3. Запускає Puppeteer (`puppeteer-core` + системний Chromium), відкриває `http://frontend:5173/characters/:id/print?print_token=...` **всередині докер-мережі** (dev: порт 5173 сервісу `frontend`; prod — уточнити, як `frontend` роздає статику там, ймовірно через `nginx`/served build, тоді URL інший).
  4. Чекає, поки сторінка повністю відрендериться (мережеві запити персонажа/заклинань завершились — `page.waitForSelector` на якийсь маркер "дані завантажено", а не голий `networkidle`, бо PWA-сервіс-воркер може заважати).
  5. Викликає `page.pdf({ format: 'A4', printBackground: true })`.
  6. Віддає бінарник з `Content-Type: application/pdf` та `Content-Disposition: attachment; filename="<ім'я персонажа>.pdf"`.

**Проблема авторизації, яку вирішує print-токен:** маршрут `/characters/:id/print` за замовчуванням вимагає звичайного логіну (кукі в браузері користувача), яких у server-side Puppeteer немає. Print-токен — одноразовий обхідний шлях: `CharacterSheetPrint.jsx` (чи його data-fetching) повинен вміти прийняти `?print_token=` замість звичайної сесії, а бекенд (`character-sheet` і, можливо, інші сервіси, чиї дані показуються на друкованому листі — `spellbook`) повинен вміти звірити цей токен як альтернативу звичайному access-токену.

### 3.3. Кнопка на сайті
У шапці `CharacterSheet.jsx` (поруч із "Поділитись"/іншими діями) — кнопка "Завантажити PDF", що б'є в `GET /api/characters/:id/pdf` (просте посилання/завантаження, браузер сам обробить `Content-Disposition: attachment`).

### 3.4. Docker
`services/character-sheet/Dockerfile` — зараз `FROM node:20-alpine`. Для Puppeteer:
- Або `apk add --no-cache chromium` + `puppeteer-core` (вказати `executablePath` на системний Chromium) — легший шлях на Alpine.
- Або перейти на `node:20-slim` (Debian) для цього сервісу і використати повний пакет `puppeteer` (сам качає сумісний Chromium) — простіше в налаштуванні, але Debian-образ важчий за Alpine базово.
- Перевірити вплив на розмір фінального образу і час білду — це реальний компроміс, який варто явно проговорити з користувачем перед імплементацією (наскільки некритичний зайвий розмір образу на self-hosted залізі).

## 4. Файли, які зачепить реалізація

- `services/frontend/src/pages/CharacterSheetPrint.jsx` — новий, друкований макет.
- `services/frontend/src/App.jsx` — новий маршрут `/characters/:id/print` (ймовірно поза звичайним Navbar/BottomNav shell).
- `services/frontend/src/pages/CharacterSheet.jsx` — кнопка "Завантажити PDF" у шапці.
- `services/character-sheet/src/routes/*.js` — новий `GET /:id/pdf`.
- `services/character-sheet/src/controllers/*.js` — новий контролер/метод для генерації PDF (перевірка прав + виклик Puppeteer-хелпера).
- `services/character-sheet/src/services/pdf.service.js` (новий файл) — інкапсулює запуск Puppeteer, побудову print-токена, виклик `page.pdf()`.
- `services/character-sheet/package.json` — нова залежність (`puppeteer-core` або `puppeteer`).
- `services/character-sheet/Dockerfile` — Chromium-залежність.
- Можливо `services/auth` або спільна утиліта — якщо print-токен генерується/валідується через існуючий JWT-механізм `auth`-сервісу, а не локально в `character-sheet`.

## 5. Відкриті питання для уточнення перед стартом реалізації

1. Обсяг "маневрів" — чи є (чи буде до моменту реалізації) окрема структура даних, чи це лишається вільним текстом.
2. Чи потрібен PDF і для **публічного** перегляду персонажа (`/characters/public/:id`) без логіну, чи лише для власника/GM.
3. Формат імені файлу / чи потрібна кирилиця в `Content-Disposition` (може вимагати `filename*=UTF-8''...` кодування).
4. Чи прийнятний розмір Docker-образу `character-sheet` зі вбудованим Chromium, чи варто одразу виносити PDF-генерацію в окремий легший/важчий сервіс.
5. Dev vs prod URL, який Puppeteer використовує для рендеру print-сторінки (в dev — `frontend:5173`, у prod build фронтенд роздається інакше — уточнити через `nginx`/`docker-compose.prod.yml`).

## 6. Верифікація (коли буде реалізовано)

1. `docker compose up --build` — перезбірка `character-sheet` з новою Chromium-залежністю, переконатись що контейнер стартує.
2. Відкрити `/characters/:id/print?print_token=...` напряму в браузері — щільний макет без табів/навбару, коректно розбитий на сторінки при друку (Ctrl+P → Save as PDF у самому браузері як швидка перевірка перед підключенням Puppeteer).
3. Натиснути "Завантажити PDF" на реальному персонажі — файл завантажується, відкривається, візуально звірити з референсними `ttrpg-system/*.png`.
4. Перевірити доступ: власник качає PDF свого персонажа; чужий приватний персонаж — 403; публічний персонаж — відповідно до відповіді на відкрите питання №2.
5. Персонаж з великою кількістю заклинань/спорядження — коректна розбивка на кілька сторінок, нічого не обрізається й не накладається.
6. Персонаж майже без даних (щойно створений майстром) — PDF не ламається на порожніх секціях.
